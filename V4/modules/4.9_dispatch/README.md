# 4.9 优化调度模块

当前 V4 是“边界—请求—响应—再调度—提交”的可行性协调器，不宣称 Pareto、随机或滚动最优。

- `model/v4_dispatch_4_9.m`：依据 4.3、4.5、4.6、4.7 边界生成第一轮请求；电解槽达到最低运行点后可继续吸收富余功率直至额定值。
- `model/v4_validate_dispatch_4_9.m`：检查容量、SOC、充放互斥、算力 0/在线最低功率互斥及计划闭合。
- `model/v4_redispatch_after_compute_4_9.m`：4.6 实际功率低于请求时，在调用 4.7 前把释放功率补给已申报海洋需求。
- `model/v4_reconcile_actuals_4_9.m`：核对模块实际响应；若可调负荷缺供与弃电并存则拒绝提交并要求再调度。
- `model/test_v4_dispatch_scenarios_4_9.m`：验证电解槽升至额定功率以及新能源骤降后“电池放电、柔性算力下降、终段无弃电”。
- `docs/4.9调度与求解说明.md`：说明协调顺序、算力模式和 72 小时验收条件。

设备效率、SOC、库存和交付能力仍由4.3—4.7各责任模块执行，4.9不得改写其物理状态。
