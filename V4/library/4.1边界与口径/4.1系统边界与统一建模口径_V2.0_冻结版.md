> 文档版本：V2.0  
> 文档状态：**FROZEN_INTERFACE（公共结构冻结）**  
> 配套模式：`BLUE_HUB_CH4_SCHEMA_V2`、`common_case_v2`  
> 上位依据：《4.1边界与口径冻结表》V1.0、《4.3—4.7模型整合与公共接口冻结建议》  
> 适用范围：4.3多源能源、4.4公共母线、4.5电池—制氢—储氢、4.6绿色算力、4.7多形态输出、4.8评价体系和4.9智能调度  
> 数据原则：未取得工程或企业数据的数值标注`[假设值，待企业调研校准]`；尚未核验来源的专业公式标注`[需查证文献支撑]`。

## 4.1.1 本节作用

本节不重复建立风机、光伏、潮流能、电池、电解槽、数据中心或海缆设备公式，而是冻结第四章联合模型的公共合同，包括空间边界、计量点、功率方向、损耗归属、时间尺度、基准方案、经济与生命周期边界以及模型分层。4.2把这些口径转化为变量字典和机器可读配置；4.3—4.7只保留本模块的物理状态、可行域和接口计算；4.4是唯一公共功率总账，4.9是唯一跨模块调度决策者。

本版完成后，空间边界、测点、单位、正负方向、损耗所有权、端口链路、执行顺序和状态提交规则不再由后续小节修改。`interface_smoke`可用于接口联调；只有P0待决策项关闭并完成项目组签认后，`engineering_base`才可用于工程经济性、消纳率、可靠性和环保性结论。P0数值的补齐只升级`parameterVersion`，不改变公共接口结构。

## 4.1.2 系统边界

### （1）纳入边界

- 4.3：漂浮式风电、漂浮式光伏、潮流能及波浪能预留接口，从设备电气端至源侧汇集点；
- 4.4：源侧汇集点之后的公共母线、公共变换和岛级公用系统功率总账；
- 4.5：构网型电池、制氢、压缩纯化和储氢内部状态及可行域；
- 4.6：算力任务队列、IT功率、冷却与辅助功率、设施总功率及柔性区间；
- 4.7：海缆外送、算力服务交付、氢能物流交付及海洋综合用能；
- 4.8—4.9：评价指标、多目标权衡、容量规划和运行调度。

### （2）边界终点

- 电力：陆上受端/结算点`MP-CABLE-RECEIVE`；
- 氢能：管道或船运损耗后的产品交付点`MP-H2-DELIVERY`；
- 算力：满足通信与服务约束后的服务交付点`MP-COMPUTE-DELIVERY`；
- 海洋用能：设备电气入口及对应服务量结算边界。

海缆受端电量、氢气最终交付量和算力交付量用于收益及生命周期评价；它们不得回接4.4替代海上送端功率，否则会提前扣除输出通道损耗。

## 4.1.3 模块责任与耦合关系

| 小节 | 唯一职责 | 向外输出 | 不得重复承担 |
|---|---|---|---|
| 4.3 | 多源设备出力、状态、预测、场景、集电损耗与互补性 | MP-03可用/接受/实际功率、源侧辅机、调节能力 | 储能调度、负荷分配、全系统功率平衡 |
| 4.4 | 公共母线实际功率守恒与损耗账本 | 平衡残差、显式弃电/缺供核验、损耗审计 | 容量优化、设备状态方程、收益决策、隐式补齐残差 |
| 4.5 | 电池、电解槽和储氢内部状态及响应可行域 | 充放电实际值、SOC、制氢功率、产氢量、库存 | 决定电力外送、算力负荷或系统弃电 |
| 4.6 | 任务队列、IT/辅助/设施功率和算力柔性 | 设施实际功率、服务量、队列、上下调能力 | 系统电源分配、重复计算外送收益 |
| 4.7 | 电、氢、算力和海洋服务的交付通道 | 送端/受端电量、氢交付、算力交付、海洋服务 | 重算制氢库存、PUE和全系统平衡 |
| 4.9 | 跨模块容量与运行决策 | 各端口requested指令 | 设备高保真动态和重复计量 |

联合运行遵循：4.3、4.5、4.6、4.7先发布当前状态下的可行边界；4.9下发`requested`；专业模块返回`accepted/actual`；4.4只以`actual`闭合母线；随后由各状态所有者提交时段末SOC、氢库存、任务队列和设备状态，4.8最后开展评价。4.4不得在残差出现时自动生成储能响应、弃电或缺供量。

统一执行阶段固定为：

```text
BOUNDARY → REQUEST → RESPONSE → LEDGER → STATE_COMMIT → EVALUATION
```

- `BOUNDARY`：4.3、4.5、4.6、4.7发布上下界、调节能力和状态；
- `REQUEST`：4.9产生跨模块请求；接口联调阶段可由明确标注的规则调度器代替；
- `RESPONSE`：各专业模块裁剪请求并返回接受值和实际值；
- `LEDGER`：4.4仅使用实际值核算功率守恒；
- `STATE_COMMIT`：4.3/4.5/4.6/4.7各自提交其拥有的下一时段状态；
- `EVALUATION`：4.8使用物理结果和交付点结果计算KPI。

## 4.1.4 公共测点与功率正方向

| 测点 | 含义 | 责任模块 |
|---|---|---|
| `MP-DEVICE` | 单台/单阵列设备电气端毛功率 | 单源模型 |
| `MP-03-SOURCE-POI` | 已扣源侧集电损耗的多源汇集功率 | 4.3 |
| `MP-04-COMMON-BUS` | 所有源、储、算、用端口的公共功率守恒点 | 4.4 |
| `MP-ELECTROLYZER` | 制氢系统电气入口 | 4.5 |
| `MP-DC-FACILITY` | 数据中心设施总用电入口 | 4.6 |
| `MP-CABLE-SEND` | 海缆海上送端 | 4.7 |
| `MP-CABLE-RECEIVE` | 海缆陆上受端/电力结算点 | 4.7 |
| `MP-H2-DELIVERY` | 氢产品最终交付点 | 4.7 |
| `MP-COMPUTE-DELIVERY` | 算力服务交付点 | 4.7 |

优化层统一使用非负端口幅值。发电、储能放电和允许的外购电为正注入；电池充电、电解槽、数据中心、海洋负荷、海缆送端、公共辅机、损耗和弃电为正消耗。电池净功率仅作为派生量：

\[
P_t^{\mathrm{BESS,net}}=P_t^{\mathrm{BESS,dis}}-P_t^{\mathrm{BESS,ch}}
\]

式中正值表示向公共母线放电。该式为本项目接口定义，不替代4.5的SOC状态方程。

每个可控端口必须保留`requested → accepted → actual`链路。`requested`来自4.9，`accepted`表示设备在当前状态下可接受的指令，`actual`表示用于4.4结算和平衡的实际测点值。

## 4.1.5 损耗与辅机唯一归属

| 项目 | 计算责任 | 总账处理 |
|---|---|---|
| 风机阵列、光伏场内电缆、潮流能集电损耗 | 4.3 | 已在MP-03上游扣除，4.4不得再扣 |
| 风/光/潮设备辅机 | 4.3计算、4.4记账 | 作为`pSourceAuxLoad`仅扣一次 |
| 公共升压、变换、母线和岛级公用系统 | 4.4 | `pPostPOILoss`和`pCommonAux` |
| 电池充放电损耗 | 4.5 | 体现在SOC方程，不作为4.4附加损耗 |
| 电解SEC、BOP、压缩纯化和储氢循环能耗 | 4.5 | 按冻结的SEC边界合成为制氢设施功率；禁止重复包含 |
| IT、冷却、泵、网络和存储辅助功率 | 4.6 | 合成为`pDCFacilityActual` |
| 海缆损耗 | 4.7 | 4.4使用送端实际功率；收益使用受端电量 |
| 管道/船运氢损耗 | 4.7 | 只减少最终交付量，不回减4.5产氢量 |
| 海洋用能设备电耗 | 4.7 | `pMarineActual`进入4.4 |

## 4.1.6 统一时间尺度

| 层级 | 冻结步长 | 主要用途 |
|---|---:|---|
| 源侧原始仿真 | 5 min | 风光潮波动、状态机和预测误差 |
| 主运营调度 | 1 h | 168 h/8760 h联合运行与经济评价 |
| 日内精细校核 | 15 min | 爬坡、柔性负荷和短时储能校核 |
| 构网动态 | 不大于1 s | 频率、电压、虚拟惯量、故障与黑启动 |
| 规划与财务 | 1 year | 建设、替换、折旧、现金流和LCA |

以上时间尺度为`[项目建模假设，待调度与设备数据校准]`。功率按能量守恒求平均，电量/产氢量/服务量求和，SOC/氢库存/任务队列取时段末状态，故障与复归事件不得线性插值，极值和爬坡指标须另行保留。

## 4.1.7 `common_case_v2`公共原型

第四章只保留一个公共参数包、两种运行模式。后续模块不得在联合运行时用自身默认值覆盖公共参数；若公共参数缺失，必须终止运行并返回明确错误：

- `interface_smoke`：采用完整但明确标注为假设的数值，用于接口、符号、单位、损耗和状态链路联调，不形成工程结论；
- `engineering_base`：用于8760小时和方案比较。场址、容量、初态、经济/LCA等P0字段未签认时必须拒绝严格运行，禁止自动回退到各模块默认值。

当前联调源侧沿用34 MW风电、4 MW光伏和2 MW潮流能，共40 MW，来源为`[现有公共基线代码假设值，待工程尺度校准]`。其余联调数值统一由`common_config_4_2.m`提供并携带假设声明。

源侧装机治理顺序为：4.3提出候选组合，4.9开展容量规划，项目组选择基准组合，4.1冻结；在运营调度方案比较中保持源侧装机一致。海缆由4.7提供技术与经济参数，但既有/新建主比较口径由4.1冻结。

## 4.1.8 基准与对比方案

“单一电力外送”和“源储算用协同”必须共用：

- 同一场址、水深、离岸距离、登陆点和公共平台拓扑；
- 同一代表年风速、辐照、潮流、海况和时间戳；
- 同一源侧装机、设备可用率、检修和故障序列；
- 同一投运年、评价期、币值年和经济/LCA边界；
- 同一随机种子和预测误差场景。

主比较采用绿地新建口径：两种方案均按同一海缆成本函数优化输出容量，用于识别多形态转化是否降低海缆容量需求。辅助比较采用既有海缆口径：既有CAPEX作为沉没成本，两种方案共用同一容量和损耗，只比较增量投资与运营结果。不得混用“单一外送新建海缆、协同方案既有海缆”。

币值基准年暂取2026年、投运年记为`Y0`、公共评价期暂取25年，均为`[假设值，待项目进度、设备寿命和融资条件校准]`。

## 4.1.9 模型分层与排除项

联合优化采用经过校准的代理、查表或分段线性模型。构网型EMT、电化学细节、漂浮平台六自由度水动力和船舶逐航次路径等由离线高保真模型校核，其结果通过收紧可行域或更新代理参数回灌，不在4.9中重复建立。

离线校核不能替代联合模型的实际功率平衡；联合模型也不能据此宣称已验证电压频率稳定性。两类结果必须分别报告。

## 4.1.11 冻结优先级与变更控制

公共规则发生冲突时，执行优先级固定为：

```text
4.1冻结边界与责任 > 4.2机器可读接口 > 4.3—4.7专业模块内部定义 > 演示脚本默认值
```

本版冻结后执行以下治理规则：

1. 4.3—4.9不得自行新增另一套公共测点、单位、端口方向或总功率平衡；
2. 后续模块若与本版不一致，修改后续模块或增加适配器，不回改4.1—4.2；
3. 场址、容量、效率、价格和初态等工程数值通过`parameterVersion`迭代，不改变`schemaId`和`schemaVersion`；
4. 只有发现会造成物理错误、法规冲突或无法表达必要端口的重大缺陷时，才允许启动新一代Schema；新Schema必须使用新编号并提供迁移表，禁止覆盖V2含义；
5. 模块内部默认值只能用于单元测试，不得进入联合运行结果；
6. 所有联合入口必须先通过`validate_common_schema_4_2`，所有模块输出必须通过`validate_module_packet_4_2`；
7. 公共标准文件发布时保存SHA-256清单，用于识别未经批准的漂移。

因此，后续P0事项关闭时只更新参数包和签认记录；`BLUE_HUB_CH4_SCHEMA_V2`的字段、单位、测点、所有权和执行阶段保持不变。
